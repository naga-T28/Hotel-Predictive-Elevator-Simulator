# シミュレーションの実験方法と結果：初心者向けガイド

## 1. この文書の目的

この文書は、`outputs/` に保存されているエレベーターシミュレーションについて、
「何を試したのか」「数字は何を意味するのか」「結果から何が言えるのか」を、
シミュレーションや統計に詳しくない人向けに説明するものである。

最初に重要な点をまとめると、現在もっとも論文用の比較実験に近い結果は
`outputs/hotel_checkout_validation/` である。この実験では、同じチェックアウト需要に対して
4つの制御条件を50回ずつ比較している。それ以外のホテル・軌跡実験は主に1回の動作確認であり、
制御方式の優劣を証明する結果ではない。

## 2. 何をシミュレーションしているか

対象は、8階建てのホテルに3台のエレベーターがある状況である。シミュレーターは、
乗客がホールへ到着し、エレベーターを待ち、乗車し、目的階へ到着するまでを再現する。

基本的なエレベーター条件は次のとおりである。

| 項目 | 設定 |
|---|---:|
| 建物 | 8階建て |
| エレベーター | 3台 |
| 1台の定員 | 15人相当 |
| 1階分の移動時間 | 2.5秒 |
| ドアを開く時間 | 1.5秒 |
| ドアを閉じる時間 | 1.5秒 |
| 1人の乗車時間 | 0.7秒 |
| 1人の降車時間 | 0.6秒 |

実験によって、乗客の発生方法、実験時間、待機位置、予測方法は異なる。正確な設定は、
各出力ディレクトリの `config.yaml` または対応する `configs/*.yaml` で確認できる。

## 3. 比較した制御方法

### 3.1 Baseline

将来の乗客情報を使わず、現在発生している呼び出しだけを見て号機を割り当てる基準方式である。
待機中の号機を特定階へ移動させる事前配置も行わない。

### 3.2 Myopic / Parking only

Myopicは、現在待っている乗客と現在の運行状態から、待ち時間が短くなる号機を選ぶ方式である。
`hotel_checkout_validation` のMyopic条件では待機位置更新も有効になっているため、
この条件は「Parking only」として扱う。

### 3.3 Nearest Car

呼び出し階に近い号機を優先する単純な方式である。分かりやすい一方、号機が今後どこへ停止するか、
将来どこから乗客が来るかは十分に考えない。

### 3.4 Predictive / Hotel predictive

現在の乗客だけでなく、近い将来に発生すると予測された乗客も前方計算に含めて号機を選ぶ。
ホテル実験では、チェックアウト時刻、宴会終了時刻、発生階、目的階などの予定情報を利用する。

### 3.5 Prescient reference

将来発生する乗客を知っているものとして計算する比較用の方式である。ただし、現在の実装では
号機割当と経路評価が近似的であるため、必ず最良になる理論上の性能上限ではない。

## 4. 実験の基本的な進め方

### 4.1 乱数シードとは

乗客の発生時刻や発生階にはランダム性がある。乱数シードは、そのランダムな乗客列を再現するための
番号である。同じ設定と同じシードを使えば、同じ乗客需要を再現できる。

方式Aと方式Bを公平に比較するには、両方へ同じシードを与える必要がある。本プロジェクトの比較実験は、
シードごとに同じ需要を各方式へ与える「対応あり比較」を行う。

### 4.2 なぜ50回実行するのか

1回だけの結果は、たまたま乗客が集中した、または需要が分散した影響を強く受ける。シードを変えて
50回実行し、その平均を取ることで、偶然による偏りを小さくする。

### 4.3 95%信頼区間とは

95%信頼区間は、繰り返し実験で得られた平均値の不確かさを表す範囲である。本結果では、
候補方式と基準方式の「待ち時間の差」もシードごとに計算している。

- 差が負：候補方式の待ち時間が短い。
- 差が正：候補方式の待ち時間が長い。
- 差の95%信頼区間がすべて0秒未満：今回の条件では、候補方式が明確に短いと判断する。
- 信頼区間が0秒をまたぐ：どちらが良いか明確とは言えない。

## 5. 評価指標の意味

| 指標 | 初心者向けの意味 | 望ましい方向 |
|---|---|---|
| 平均待ち時間 | 全乗客がホールで待った時間の平均 | 小さい方がよい |
| 待ち時間中央値 | 半数の乗客がこの時間以内に乗れたことを示す | 小さい方がよい |
| 95%待ち時間 | 95%の乗客がこの時間以内に乗れたことを示す | 小さい方がよい |
| 最大待ち時間 | 最も長く待った乗客の待ち時間 | 小さい方がよい |
| 平均乗車時間 | 乗車してから目的階へ着くまでの平均時間 | 小さい方がよい |
| 平均総サービス時間 | ホール到着から目的階到着までの平均時間 | 小さい方がよい |
| 30秒超待ち率 | 30秒より長く待った乗客の割合 | 小さい方がよい |
| 乗り残し率 | 満員などで乗車できなかった乗客の割合 | 小さい方がよい |
| 空運転距離 | 乗客を乗せずに走った距離 | 小さい方が効率的 |
| 総走行距離 | 乗客の有無にかかわらず走った合計距離 | 小さい方が効率的 |
| 停止回数 | エレベーターが停止した回数 | 少ない方が一般に効率的 |

待ち時間だけが短くても、空運転距離が大幅に増えれば、利便性と運行効率の間にトレードオフがある。
そのため、複数の指標を一緒に読む必要がある。

## 6. 論文再現条件の結果

### 6.1 Predictive方式を50回実行した結果

`outputs/paper_reproduction/summary.json` は、8階・3号機・下りピーク交通において、
Predictive方式を50シードで実行した集計である。

| 指標 | 50回平均 | 95%信頼区間 |
|---|---:|---:|
| 平均待ち時間 | 18.21秒 | 18.02～18.40秒 |
| 待ち時間中央値 | 17.10秒 | 16.89～17.30秒 |
| 95%待ち時間 | 36.30秒 | 35.84～36.76秒 |
| 最大待ち時間 | 56.90秒 | 55.03～58.77秒 |
| 平均総サービス時間 | 44.03秒 | 43.54～44.51秒 |
| 30秒超待ち率 | 12.97% | 12.46～13.49% |
| 乗り残し率 | 0% | 0～0% |
| 1時間当たり輸送人数 | 455.94人 | 450.96～460.92人 |

平均待ち時間のばらつきは比較的小さく、設定した約453.6人/時の需要をおおむね処理できている。
一方、参考論文で報告されたPredictive方式の15.3秒より今回の18.21秒は長いため、参考論文と
同じ改善性能を再現したとは言えない。

### 6.2 4方式を比較した予備結果

`outputs/paper_reproduction/comparison_summary.json` は4方式を比較しているが、各方式5回であり、
50回比較ではない。したがって、これは予備結果として扱う。

| 方式 | 平均待ち時間 | 平均待ち時間の95%信頼区間 | 95%待ち時間 |
|---|---:|---:|---:|
| Myopic | **17.65秒** | 16.90～18.41秒 | **36.14秒** |
| Nearest Car | 18.10秒 | 17.42～18.79秒 | 37.55秒 |
| Prescient reference | 17.92秒 | 17.18～18.67秒 | 36.76秒 |
| Predictive | 18.26秒 | 17.54～18.98秒 | 36.83秒 |

この5回比較ではPredictive方式はMyopic方式より平均0.61秒長く、差の95%信頼区間は
0.34～0.87秒だった。つまり、この下りピーク条件では予測方式の改善は確認できなかった。
ただし実行回数が少ないため、論文の最終比較には各方式50回以上の実験が必要である。

## 7. チェックアウト需要の50回比較

### 7.1 実験方法

`outputs/hotel_checkout_validation/` は、客室階からロビーへチェックアウト客が集中する条件を、
次の4条件で50回ずつ比較した結果である。

1. Baseline：将来情報なし、事前配置なし。
2. Myopic（Parking only）：将来情報なし、ロビー待機あり。
3. Hotel predictive：ロビー待機に加え、ホテル予定を使った将来需要予測あり。
4. Prescient reference：ロビー待機に加え、実際の将来到着情報を参照。

すべての条件で同じ50個のシードを使っているため、各方式には同じ乗客需要が与えられている。

### 7.2 利用者側の結果

| 条件 | 平均待ち時間 | 95%待ち時間 | 平均総サービス時間 | 30秒超待ち率 | 乗り残し率 |
|---|---:|---:|---:|---:|---:|
| Baseline | 14.98秒 | 29.30秒 | 33.44秒 | 4.68% | 0% |
| Parking only | 14.98秒 | 29.30秒 | 33.44秒 | 4.68% | 0% |
| Hotel predictive | **13.81秒** | **25.19秒** | **30.34秒** | **2.43%** | 0% |
| Prescient reference | 14.42秒 | 27.82秒 | 31.88秒 | 3.63% | 0% |

Hotel predictiveはParking onlyより平均待ち時間を1.17秒、割合では7.80%短縮した。
シードごとの対応差の95%信頼区間は−1.31～−1.03秒で、範囲全体が0秒未満である。
したがって、今回設定したチェックアウト条件では、ホテル情報を使った予測により平均待ち時間が
明確に短縮されたと判断できる。

95%待ち時間も29.30秒から25.19秒へ約4.11秒短縮し、30秒超待ち率は4.68%から2.43%へ
減少した。平均値だけでなく、比較的長く待つ利用者も減ったことを示している。

BaselineとParking onlyは完全に同じ結果だった。チェックアウト客はロビーへ移動するため、
乗客を降ろした号機が自然にロビーへ集まり、追加のロビー待機指示が実質的な差を生まなかったためと
考えられる。

### 7.3 運行効率の結果

| 条件 | 空運転距離 | 総走行距離 | 停止回数 |
|---|---:|---:|---:|
| Baseline | **593.76階** | **1,286.00階** | **349.72回** |
| Parking only | **593.76階** | **1,286.00階** | **349.72回** |
| Hotel predictive | 708.54階 | 1,460.12階 | 377.48回 |
| Prescient reference | 651.18階 | 1,378.60階 | 363.96回 |

Hotel predictiveは待ち時間を短くした一方、Baselineより空運転距離が約19%、総走行距離が約14%、
停止回数が約8%増えた。これは、乗客を早く迎えに行くために追加の移動や停止を行ったことを示す。

したがって、この結果は「何の負担もなく待ち時間を短縮した」という意味ではない。利用者の待ち時間と、
走行量・エネルギー消費の間にトレードオフがある。現在はエネルギー消費量を直接計算していないため、
実用性を評価するには今後の追加実験が必要である。

### 7.4 Prescient referenceが最良でない理由

Prescient referenceは将来情報を知っているが、現在の実装では呼び出しが発生するたびに号機を選ぶ
逐次的な方式であり、すべての号機経路をまとめて最適化しているわけではない。そのため、将来情報を
持つだけで必ず最良になるとは限らない。この結果は「完全な将来情報が役に立たない」ことではなく、
将来情報の利用方法や前方計算の近似に改善の余地があることを表している。

## 8. その他の単発シナリオ

以下は各シナリオを1シードで実行した結果である。実行条件や実験時間が異なるため、シナリオ間で
単純に優劣を付けることはできない。また、各シナリオでBaselineとの比較をしていないため、
予測制御による改善量も判断できない。

| シナリオ | 平均待ち時間 | 95%待ち時間 | 最大待ち時間 | 30秒超待ち率 | 乗り残し |
|---|---:|---:|---:|---:|---:|
| チェックアウト単発 | 14.35秒 | 25.32秒 | 50.22秒 | 2.99% | 0人 |
| 朝食時間帯 | **10.54秒** | 28.42秒 | 45.18秒 | 4.28% | 5人 |
| 宴会終了 | 12.73秒 | 29.90秒 | **42.42秒** | 5.30% | 0人 |
| 軌跡予測 | 19.21秒 | 40.10秒 | 109.41秒 | 15.82% | 0人 |

朝食時間帯では平均待ち時間は短いが、309人中5人、約1.62%の乗り残しが発生した。
平均待ち時間だけでは、この問題を見落とす可能性がある。

軌跡予測では最大待ち時間が109.41秒、60秒超待ち率が約1.32%となった。全員を最終的に輸送できたが、
一部の乗客に長時間待ちが集中している。予測なし、軌跡予測、正確な予測を同じ需要で複数回比較するまで、
軌跡予測の有効性は判断できない。

## 9. `outputs/` 内のファイルの読み方

| ファイル | 内容 |
|---|---|
| `config.yaml` | 実際に使った建物、交通、予測、シミュレーション設定 |
| `summary.json` | 1回の実験、または反復実験の主要な集計結果 |
| `comparison_summary.csv/json` | 制御条件ごとの平均、標準偏差、95%信頼区間 |
| `paired_comparisons.csv/json` | 同じシード同士で比較した待ち時間差と改善率 |
| `passenger_metrics.csv` | 乗客ごとの待ち時間、乗車時間、状態 |
| `elevator_metrics.csv` | 号機ごとの走行距離、空運転距離、停止回数、稼働時間 |
| `event_log.csv` | 乗車、降車、移動、ドア動作などの時系列ログ |
| `assignment_log.csv` | 各ホール呼び出しをどの号機へ割り当てたか |
| `runs/seed_*/` | 反復実験の各シードに対応する未集計結果 |
| `figures/` | 待ち時間分布、号機位置、乗車人数などのグラフ |

JSONはプログラムで読みやすい形式、CSVは表計算ソフトで開きやすい形式である。初心者が結果を
確認するときは、まず `comparison_summary.csv` と `paired_comparisons.csv` を見るとよい。

## 10. 現時点で言えること

結果から妥当に言えることは次のとおりである。

- 今回設定したチェックアウト需要では、Hotel predictiveが非予測方式より平均待ち時間を
  約1.17秒、7.80%短縮した。
- 同じ条件で、95%待ち時間と30秒超待ち率も改善した。
- その代わり、空運転距離、総走行距離、停止回数は増加した。
- 下りピークの5回比較では、Predictive方式はMyopic方式を上回らなかった。
- 朝食、宴会、軌跡予測については動作結果があるが、制御方式の優劣を判断できる比較実験は未完了である。

結果からまだ言えないことは次のとおりである。

- すべてのホテルや交通条件で予測型制御が有効であること。
- 実際のホテルでも同じ待ち時間削減率が得られること。
- Prescient referenceが理論上の最適解であること。
- 待ち時間の短縮が電力消費や保守費用を含めても有利であること。
- 論文で提案した多目的Rolling Horizon Optimizationを完全に検証したこと。

## 11. 今後必要な実験

論文として主張を強くするには、次の実験が必要である。

1. 朝食と宴会についても、Baseline、Parking only、Hotel predictive、Prescient referenceを
   同じ50シードで比較する。
2. 宴会予定時刻を0、±1分、±3分、±5分ずらし、予測誤差への強さを確認する。
3. 予測時間10、20、40、60秒を同じ需要で比較する。
4. 荷物係数を変え、実質的な定員低下への影響を確認する。
5. 人数予測と目的階予測の誤差を予測側だけに加えて比較する。
6. 割当計算時間とエネルギー消費の近似値を記録する。
7. 可能であれば、匿名化された実ホテルの人流・運行データでモデルを校正する。

具体的な実験計画と判定基準は `EXPERIMENT_VALIDATION_PLAN.md` を参照する。

## 12. まとめ

現在の結果の中では、チェックアウト需要の4条件×50回比較が最も信頼できる。その結果、
ホテル予定を使う予測型制御は、今回の条件では平均待ち時間と長時間待ちを減らした。一方で、
エレベーターの走行量と停止回数は増加した。

したがって、本シミュレーションは「ホテル情報が待ち時間短縮に役立つ可能性」を示しているが、
「あらゆるホテルで有効」「運行コストも含めて最適」と結論づける段階ではない。朝食、宴会、
予測誤差、荷物、予測時間について同じ方法で比較を追加することで、論文としてより妥当な評価になる。
