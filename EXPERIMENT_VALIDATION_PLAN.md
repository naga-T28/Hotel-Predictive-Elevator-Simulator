# 添付論文に対応した実験修正計画

## 1. 位置付け

現在の出力は、ホテル固有の交通を生成し、予測型群管理を動作させられることを示す
概念実証である。一方、添付論文が主張する「ホテル固有データを利用する効果」と
「待機階だけでなく号機割当を予測最適化する効果」を検証するには、同一需要に対する
対照実験と統計的比較が必要である。

修正後の実験では、次の仮説を検証対象とする。

- H1: ホテル情報利用型予測制御は、将来情報を使わない制御より平均待ち時間を短縮する。
- H2: 予測制御は、待機位置の変更だけを行う方式より平均待ち時間を短縮する。
- H3: 予測誤差が増えると効果は低下するが、一定範囲では非予測型制御を上回る。
- H4: 平均待ち時間の改善が、95%待ち時間、乗り残し率、空運転距離の著しい悪化を伴わない。

## 2. 必須の比較条件

チェックアウト、朝食、宴会終了の各シナリオで、同一の乗客到着系列を使って次の条件を
比較する。各方式には同じ50個の乱数シードを使用する。

| 条件 | 呼び出し割当 | 事前配置 | 将来情報 | 目的 |
|---|---|---|---|---|
| Baseline | Myopic | なし | なし | 従来方式相当の基準 |
| Parking only | MyopicまたはNearest Car | あり | なし | 事前配置だけの効果 |
| Hotel predictive | Predictive | Parking onlyと同一 | ホテル予定 | ホテル情報の追加効果 |
| Prescient | Prescient | Hotel predictiveと同一 | 完全情報 | 達成可能な上限の参考 |

Parking onlyとHotel predictiveでは、待機戦略、再配置間隔、エレベーター仕様、需要系列を
すべて同一にする。これを守らないと、予測と待機位置のどちらが結果を改善したか判別できない。

## 3. シナリオ別条件

### 3.1 チェックアウト

- 客室階からロビーへの150人の集中需要を使用する。
- Baseline、ロビー待機、客室階へのゾーン配置、ホテル予測を比較する。
- 荷物係数1.0、1.5、2.0を追加し、実質容量低下に対する頑健性を確認する。

### 3.2 朝食

- 客室階から2階レストランへの移動と、2階から客室階・ロビーへの移動を含める。
- 全方式で同じゾーン配置を使用する比較を設ける。
- 開始需要と終了需要を時間帯別にも集計し、全時間平均によってピークが隠れないようにする。

### 3.3 宴会終了

- 3階から120人が300秒程度に集中する需要を使用する。
- 実需要のイベント時刻は固定し、予測器に渡す予定時刻だけを0、±60、±180、±300秒ずらす。
- 正負の誤差を別々に実行する。早い予測と遅い予測では運行への影響が異なるため、絶対値だけでまとめない。

### 3.4 予測時間

添付論文で提案している40〜60秒を検証するため、10、20、40、60秒を比較する。
予測時間以外の設定と需要系列は固定する。

## 4. 評価・統計処理

主要評価指標は平均待ち時間とする。副次指標として以下を報告する。

- 待ち時間の中央値と95パーセンタイル
- 30秒超・60秒超待ち率
- 乗り残し率
- ホール到着から目的階到着までの総サービス時間
- 総走行距離、空運転距離、停止回数
- 号機別稼働率および占有率
- 1回の割当計算に要する時間（実運用可能性の評価時）

各指標について平均、標準偏差、95%信頼区間を示す。主要比較では、共通乱数シードを
用いたシードごとの対応差とその95%信頼区間を報告する。平均待ち時間の改善率は
`(Baseline - Candidate) / Baseline * 100` とする。複数のシナリオを同時に検定する場合は、
多重比較の影響も明記する。

## 5. 実行方法

`compare` は、`--runs` を省略するとYAMLの `experiment.repetitions` を使用する。
最初に指定した方式が既定の基準方式となり、方式別集計に加えて対応あり比較を出力する。

```bash
python -m elevator_sim compare \
  --config configs/paper_reproduction.yaml \
  --schedulers myopic,nearest_car,predictive,prescient \
  --baseline myopic \
  --runs 50 \
  --output outputs/paper_reproduction_validation
```

主な集計ファイルは以下である。

- `comparison_summary.csv`: 方式ごとの平均、標準偏差、95%信頼区間
- `paired_comparisons.csv`: Myopicとの差、差の95%信頼区間、改善率
- 各方式のサブディレクトリ: 代表シード、および `runs/seed_*` に全反復のログ

ホテル実験では、事前配置なしと事前配置ありを別の設定として実行する。その際、
`hotel_events`、シミュレーション時間、基礎交通量、乱数シードは変更しない。

## 6. 論文で主張できる判定基準

Hotel predictiveと対応する対照方式の平均待ち時間差について、95%信頼区間の上限が0秒未満
であれば、今回のシミュレーション条件では予測制御が待ち時間を短縮したと判断する。
信頼区間が0秒を含む場合は「明確な差は確認できなかった」とする。待ち時間が改善しても、
乗り残し率または空運転距離が大きく悪化した場合は、利便性と運行効率のトレードオフとして
報告し、単純に有効とは結論づけない。

## 7. 現実装と論文記述の整合性

現実装のPredictive Schedulerは、各呼び出しの号機割当時に、予測乗客を含む前方計算の
平均待ち時間を評価する。添付論文の式(1)に含まれる95%待ち時間、混雑度、停止回数、
空運転距離を統合した目的関数や、割当済み呼び出しの再割当、停止順序全体の最適化は
まだ実装していない。したがって、現段階では「Rolling Horizon Optimizationの完全実装」
ではなく「予測情報を用いた逐次号機割当と定期的な待機位置更新」と記述する。

また、ホテル情報予測は需要生成時と同じイベント設定を参照するため、誤差を導入しない条件は
予定が正確な理想条件に近い。実ホテルでの有効性を論じる前に、予定時刻、人数、目的階分布の
誤差を予測側だけに加える実験が必要である。

## 8. 完了条件

- 各ホテルシナリオについて4条件を同じ50シードで実行している。
- 対応差、改善率、95%信頼区間が保存されている。
- 宴会時刻の0、±1、±3、±5分条件を実行している。
- 10、20、40、60秒の予測時間を比較している。
- 論文本文が実装済みの制御範囲を超えて主張していない。
- 実験設定、コードのコミット識別子、乱数シード、未集計ログを保存している。
